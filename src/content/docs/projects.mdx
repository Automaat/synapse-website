---
title: Projects & Git
description: How Sybra mirrors GitHub repos and manages per-task git worktrees.
---

Projects connect tasks to real codebases. Sybra clones repos as bare repositories and creates isolated worktrees for each agent.

## Storage layout

```
~/.sybra/
├── projects/          # YAML metadata per project
│   └── owner-repo.yaml
├── clones/            # bare git repos
│   └── owner-repo.git/
└── worktrees/         # per-task checkouts
    └── task-abc123/
```

## Creating a project

```sh
sybra-cli project create --url https://github.com/owner/repo
```

Or via the GUI: **Projects → Add Project → paste GitHub URL**.

Sybra clones the repo as a bare repository into `~/.sybra/clones/`. The bare clone is the source of truth; worktrees branch off it.

## project_id on tasks

Link a task to a project:

```yaml
project_id: owner/repo
```

When the agent starts, Sybra automatically:

1. Creates a new branch `sybra/<task-id>` off the default branch
2. Checks it out as a worktree at `~/.sybra/worktrees/<task-id>/`
3. Sets the agent's working directory to the worktree

On agent completion (or manual cleanup), the worktree is removed. The branch remains for you to review and merge.

## Keeping clones fresh

Sybra runs a background fetch on registered projects. You can also trigger manually:

```sh
sybra-cli project fetch owner/repo
```

## project_types

Each machine can declare which project types it handles:

```yaml
# ~/.sybra/config.yaml
project_types: [pet, oss]
```

Project types are a free-form string you assign when creating a project (`--type pet`). Automations (GitHub Issues, Todoist, Renovate) filter by `project_types` so multi-machine setups don't duplicate work.

| Machine | `project_types` | Handles |
|---------|----------------|---------|
| Laptop | `[work]` | Work projects |
| Home server | `[pet, oss]` | Personal + open-source |

---
title: Agent Modes
description: Headless vs. interactive agents — when to use each and how they work.
---

Sybra supports two execution modes, selectable per-task via `agent_mode`.

## Headless

```yaml
agent_mode: headless
```

Spawns `claude -p "<task body>" --output-format stream-json` as a subprocess. Sybra reads stdout line-by-line and streams `StreamEvent` objects to the GUI.

**Use when:**
- The task is well-defined and self-contained
- You want fully autonomous execution
- You're running multiple agents in parallel
- The task is on a remote server with no display

**Stream event types:**

| Type | Description |
|------|-------------|
| `init` | Session started, session ID assigned |
| `assistant` | Text response from Claude |
| `tool_use` | Tool call (Read, Edit, Bash, …) |
| `tool_result` | Output from a tool call |
| `result` | Final result with cost/turn stats |

**Resuming sessions:**

Sybra stores the session ID from the `init` event. If you restart a headless agent on the same task, it resumes the conversation:

```sh
claude -p "continue" --resume <session-id> --output-format stream-json
```

## Interactive

```yaml
agent_mode: interactive
```

Opens an interactive Claude Code session you can watch and steer in real time.

**Use when:**
- You want to watch the agent work and intervene
- The task requires tool approvals you want to handle manually
- You're debugging or exploring the codebase alongside the agent

Click **Attach** in the GUI to open the interactive session in your terminal.

## Parallel execution

Multiple headless agents run concurrently — Sybra doesn't impose a limit. Each gets its own process, worktree, and event stream. The GUI shows all active agents in the sidebar.

:::tip
For large batches, consider the memory footprint: each headless agent uses a full `claude` process plus its worktree checkout.
:::
